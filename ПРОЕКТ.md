### Цифровая система для fashion-бизнеса в Telegram

## Проблема

Елена ведёт успешный fashion-бизнес через Telegram. Её сила — живая персональная продажа, эмоциональный контакт с клиентом, стилистическая экспертиза. Но весь операционный процесс держится на ручной переписке:

- заказы собираются из десятков сообщений и голосовых
- профили клиентов — только в голове продавца
- статусы и треки — только если клиент сам спросит
- возвраты — длинные эмоциональные переговоры
- визит в шоурум — разовая продажа, данные теряются
- уведомления — только через Telegram-канал (ненадёжно)

Бизнес упёрся в потолок ручного управления.

---

## Решение

Система, которая **не заменяет** живую продажу, а **добавляет структуру** поверх неё:

```
Telegram — живое общение, эмоции, витрина
Система  — каталог, заказ, профиль, статусы, уведомления
```

## Цели проекта

### Бизнес-цели

- Снизить время на ведение одного заказа в 2–3 раза
- Уменьшить количество возвратов из-за ошибок по размеру/цвету
- Увеличить повторные продажи через персональные уведомления
- Превратить каждый шоурум-визит в долгосрочный источник продаж
- Снизить эмоциональную нагрузку на Елену

### Продуктовые цели

- Зафиксировать заказ в одном месте (не в переписке)
- Дать клиенту прозрачность по статусу его заказа
- Хранить профиль клиента с параметрами и предпочтениями
- Реализовать персональные уведомления не через Telegram
- Создать примерочный вишлист для клиентов шоурума

---

## Объём системы

### Войдёт в проект

- Каталог товаров (классика + новинки + фильтры)
- Профиль клиента (параметры, предпочтения, события, личные фразы)
- Карточка заказа (позиции, статусы, оплата)
- Сценарий частичной отправки (несколько посылок)
- Сценарий возврата с автоматическим расчётом суммы
- Вишлист (онлайн / примерка / ожидание поступления)
- Сессия примерки в шоуруме
- Персональные уведомления (email, WhatsApp, Telegram-бот)
- Рекомендации похожих товаров по вишлисту
- ИИ-черновики тёплых персональных сообщений
- Загрузка товара от поставщиков (ручная + CSV-импорт)

### Не войдёт в первую версию

- Интеграция с платёжными системами (онлайн-оплата)
- Интеграция с остатками поставщиков в реальном времени
- Мобильное приложение
- Полноценная аналитика и отчёты

---

## Ключевые принципы

1. **Не ломать живую продажу.** Telegram остаётся каналом общения.
2. **Голос Елены.** Автоматические сообщения звучат как она, не как робот.
3. **Клиент объясняет себя один раз.** Профиль хранит всё.
4. **Простота для Елены.** Интерфейс без обучения.
5. **MVP сначала.** Проверять гипотезы, потом расширять.

---

## Риски

| Риск | Вероятность | Как снизить |
|---|---|---|
| Елена не захочет работать в новой системе | Средняя | Вовлечь её на этапе проектирования UI |
| Клиенты не будут заходить в личный кабинет | Средняя | Сделать вход простым (ссылка в сообщении) |
| Данные о клиентах сложно перенести из переписки | Высокая | Начать с новых клиентов, старых переносить постепенно |
| Поставщики не дают структурированные данные | Высокая | Сделать удобный ручной ввод как основной |
| Разработка затянется | Средняя | Жёсткий MVP-scope, no-code для первого этапа |

## Технологии (предварительно)

### Вариант A — No-code (быстрый старт)

| Инструмент | Для чего |
|---|---|
| Notion | База данных клиентов и заказов |
| Tally / Typeform | Форма запроса, анкета клиента |
| Make / Zapier | Автоматизация уведомлений |
| Softr / Super | Клиентский личный кабинет поверх Notion |

Плюсы: запуск за 2–3 недели, без разработчика
Минусы: ограничения гибкости, не масштабируется

### Вариант B — Разработка (правильный путь)

| Слой | Технология |
|---|---|
| Frontend | Next.js (React) |
| Backend | Node.js / Rails |
| База данных | PostgreSQL |
| Уведомления | SendGrid (email) + Wati (WhatsApp) |
| ИИ-черновики | Claude API |
| Хостинг | Vercel + Railway / Render |

Плюсы: полный контроль, масштабируется
Минусы: 6–10 недель, нужен разработчик

### Рекомендация

Начать с Варианта A — проверить, что Елена и клиенты пользуются системой.
Параллельно проектировать Вариант B для полноценной разработки.
