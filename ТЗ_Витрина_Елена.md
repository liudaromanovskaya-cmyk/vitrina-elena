## 1. Суть проекта

### Ключевая проблема в одной фразе

У Елены сильная живая персональная продажа и лояльные клиенты, но весь бизнес держится на длинной ручной переписке без единой фиксации заказа, статусов, профилей клиентов и остатков. Из-за этого растут путаница, перегруз, возвраты и потеря управляемости.

### Ключевая гипотеза

Если не ломать Telegram-формат Елены, а поверх него добавить структуру — каталог, профиль клиента, карточку заказа, статусы, вишлист и персональные уведомления — можно сохранить её главную силу (живую продажу) и при этом убрать операционный хаос.

### Чего система НЕ должна делать

- Заменять живое общение Елены с клиентами
- Превращать бизнес в безликий интернет-магазин
- Убирать тепло и личность из коммуникации
- Требовать от клиентов сложного обучения

### Чего система ДОЛЖНА достичь

- Зафиксировать заказ в одном месте
- Сделать статусы прозрачными для клиента
- Хранить профиль клиента (размеры, предпочтения, события)
- Уведомлять клиентов персонально и тепло
- Снизить количество возвратов и конфликтов
- Разгрузить Елену от роли CRM, склада и бухгалтерии одновременно

## 2. Бизнес AS-IS

### Как устроен процесс сейчас

```
Клиент видит пост в Telegram-канале (фото, видео, описание)
        ↓
Переходит в личку → начинается консультация
(рост, размер, цвет, фасон, повод, длина, состав)
        ↓
Подбор вручную — по памяти продавца, через голосовые и текст
        ↓
Заказ "собирается" из десятков сообщений, пересылок, голосовых
Нет единой точки. Список временный, постоянно меняется.
        ↓
Елена делает ручной список с ценами (ещё не финальный)
        ↓
Оплата переводами на разные номера телефона (иногда на родственников)
Иногда двумя частями, иногда разными суммами
        ↓
Отправка — частичная или полная
Клиент часто не знает заранее, что посылка будет неполной
        ↓
Клиент сам проверяет состав, сверяет с переписью, пишет об ошибках
        ↓
Возврат / перерасчёт — долгая переписка, ручная сверка,
поэтапный возврат денег, эмоциональная нагрузка
        ↓
Процесс формально не закрывается — тянется хвостами
```

### Роли Елены — слишком много на одном человеке

| Роль | Что делает |
|---|---|
| Продавец | Показывает товар, дожимает сделку |
| Стилист | Подбирает по параметрам, советует |
| Менеджер | Ведёт заказ через переписку |
| Складской координатор | Знает, что есть, чего нет, что ждать |
| Бухгалтер | Считает суммы, принимает оплаты |
| Служба доставки | Организует отправку, сообщает треки |
| Служба возврата | Принимает претензии, считает возвраты |

### Системные дыры AS-IS

- Нет объекта "заказ" — он существует только как сумма сообщений
- Нет объекта "клиент" — профиль живёт только в голове продавца
- Нет статусов — ни клиент, ни продавец не знают "где мы сейчас"
- Нет фиксации того, что подтверждено
- Нет прозрачности по частичным отправкам
- Нет процесса возврата — только переговоры
- Финансовый контур не централизован

## 3. Бизнес TO-BE

### Как должен выглядеть процесс

```
Клиент видит пост в канале / ссылку на каталог
        ↓
Новый клиент → заполняет анкету (1 раз): рост, размер, цвет, повод
Постоянный клиент → заходит в личный кабинет, видит свои вещи
        ↓
Форма запроса: что ищу, под какой повод, бюджет
        ↓
Елена видит структурированный запрос → делает подборку из каталога
→ отправляет клиенту ссылку "Подборка для вас"
        ↓
Клиент подтверждает позиции → создаётся карточка заказа
(список зафиксирован: размер, цвет, цена, скидка)
        ↓
Клиент получает реквизиты → статус "оплачен"
        ↓
Елена комплектует → отмечает, что уходит в посылку 1, что в посылку 2
        ↓
Клиент получает уведомление: состав посылки + трек-номер
        ↓
Клиент видит в кабинете: что пришло, что ещё в пути
        ↓
После получения: отмечает "получено" или "хочу вернуть X"
        ↓
Возврат через форму → сумма считается автоматически → статус
        ↓
Заказ закрыт. Профиль клиента обновлён.
```

### Ключевой принцип

Telegram остаётся каналом живого общения и витриной эмоций.
Все операционные действия (зафиксировать, подтвердить, отследить, вернуть) — в системе.

## 4. MVP

### Что входит в первый запуск

| Модуль | Зачем |
|---|---|
| Каталог (классика + новинки) | Клиент видит структуру, не тонет |
| Профиль клиента с параметрами | Не объяснять размер каждый раз |
| Карточка заказа + позиции | Заказ зафиксирован в одном месте |
| Статусы заказа | Клиент знает, где он сейчас |
| Ссылка на заказ для клиента (чтение) | Прозрачность без лишней переписки |
| Форма запроса / подбора | Вход без хаоса в личке |
| Вишлист (базовый) | Клиент сохраняет, Елена видит интерес |

### Что НЕ входит в MVP (следующие этапы)

- Автоматические уведомления (сначала — вручную)
- Интеграция с платёжными системами
- Интеграция с остатками поставщиков в реальном времени
- Личный кабинет клиента (полный)
- ИИ-черновики сообщений
- Аналитика и отчёты

### Технология для MVP

**Вариант A (быстро, no-code):** Notion + Tally Forms + Zapier + Make
Плюсы: запуск за 2–3 недели, не нужен разработчик
Минусы: ограничения гибкости, не масштабируется

**Вариант B (правильно, с разработчиком):** Next.js / Rails + PostgreSQL + простой хостинг
Плюсы: полный контроль, масштабируется, можно расширять
Минусы: 4–8 недель, нужен разработчик

Рекомендация: начать с Варианта A для проверки гипотезы, параллельно проектировать Вариант B.

## 6. Поставщики

### Откуда берётся товар

Елена закупает у нескольких поставщиков / складов. Товар приходит:
- со склада в виде таблицы (Excel / Google Sheets) с артикулами и ценами
- через Telegram — фото, видео, голосовые с описанием
- через WhatsApp — аналогично
- через сайт поставщика (если есть)

### Способы загрузки товара в систему

| Способ | Когда использовать |
|---|---|
| Ручной ввод через форму | Поставщик прислал фото и видео без таблицы |
| Импорт CSV | Поставщик прислал таблицу |
| Полуавтоматический парсинг | Поставщик имеет сайт с каталогом |

MVP: ручной ввод + импорт CSV.

### Что важно при загрузке нового товара

1. Указать поставщика
2. Указать артикул поставщика (чтобы при заказе у него было легко найти)
3. Прикрепить фото и видео
4. Указать доступные размеры и цвета
5. Указать длину изделия и рекомендованный рост
6. Отметить тип товара: классика / новинка / сезонное
7. Указать теги повода и стиля (для рекомендаций)
8. Отметить похожие товары из уже существующих в каталоге

```
├── Категории

│   ├── Юбки

│   ├── Палантины и аксессуары
│   └── Комплекты
│
├── Фильтры
│   ├── Размер
│   ├── Цвет (с цветовыми чипами)
│   ├── Страна производства
│   ├── Состав ткани
│   ├── Длина (мини / миди / макси / в сантиметрах)

└── Сортировки
    ├── Новинки сначала
    ├── По цене (↑↓)
    ├── Популярные (по times_ordered)
    └── Акционные сначала
```

### Карточка товара — что показывать клиенту

- Фото (галерея, можно листать)
- Видео (если есть)
- Название и категория
- Страна производства
- Состав ткани
- Доступные размеры (недоступные — серые)
- Цвета (выбор чипами)
- Длина изделия в сантиметрах
- Рекомендованный рост
- Комментарий по посадке ("свободный крой, подходит на размер больше")
- Цена (обычная / акционная)
- Статус наличия
- Кнопка "Добавить в вишлист"
- Кнопка "Написать Елене" (переход в Telegram)

## 8. Профиль клиента

### Как создаётся профиль

**Новый клиент:** заполняет анкету через форму (ссылка от Елены или с сайта)
**Постоянный клиент:** Елена создаёт/обновляет на основе переписки
**После визита в шоурум:** обновляется автоматически из сессии примерки

### Система статусов заказа

```
новый запрос
    ↓
подбор (Елена работает над выборкой)
    ↓
согласование (клиент смотрит, уточняет)
    ↓
подтверждён ← список зафиксирован, больше не меняется
    ↓
ожидает оплату
    ↓
частично оплачен ←──────────────────┐
    ↓                                │
оплачен полностью                    │ (приходит вторая часть оплаты)
    ↓                                │
комплектуется                        │
    ↓                                │
отправлен частично ──────────────────┘
    ↓
отправлен полностью
    ↓
доставлен
    ↓
завершён ────── (без претензий)
    │
    ↓ (если есть возврат)
возврат оформлен → возврат получен → деньги возвращены → завершён
```

### Статусы позиции заказа

| Статус | Значение |
|---|---|
| `подтверждена` | Клиент сказал "беру" |
| `под вопросом` | Ещё думает |
| `в резерве` | Ожидает поставки со склада |
| `отменена` | Клиент отказался |
| `замена` | Исходная позиция заменена на другую |
| `отправлена` | Ушла в посылке |
| `доставлена` | Клиент получил |
| `возврат` | Клиент отправил обратно |

## 10. Частичная отправка

### Сценарий

```
1. Елена создаёт "Посылку 1" в карточке заказа
2. Выбирает, какие позиции уходят в ней
3. Вводит трек-номер + дату отправки
4. Система формирует сообщение клиенту:

   "Людмила, добрый день!
    Посылка 1 отправлена. Трек: RU123456789 (СДЭК).
    В ней: Платье Art-214 (беж, 46), Юбка Art-089 (бордо, 46).
    Кардиган Art-31 отправим отдельно — ориентировочно 15 декабря.
    Отслеживайте посылку: [ссылка]"

5. Клиент видит в кабинете: что пришло, что ещё в пути
6. Когда отправляется посылка 2 — повторяется шаг 4
```

### Что клиент видит в кабинете

```
МОЙ ЗАКАЗ ORD-2023-012

Посылка 1 — доставлена 01.12.2023
  ✓ Платье Art-214 (беж, 46)
  ✓ Юбка Art-089 (бордо, 46)
  Трек: RU123456789

Посылка 2 — ожидается ~15.12.2023
  ⏳ Кардиган Art-31 (молочный, 46)
```

## 11. Возврат

### Сценарий

```
1. Клиент сообщает, что хочет вернуть позицию X
2. Елена (или клиент через форму) создаёт Return:
   - указывает позицию
   - указывает причину (размер / цвет / качество / передумала)
3. Система автоматически считает сумму к возврату
4. Клиент получает адрес для отправки обратно
5. Клиент отмечает "отправила обратно" → Елена видит
6. Елена получает посылку → отмечает "получила"
7. Елена переводит деньги → отмечает "возвращено"
8. В карточке заказа всё закрыто, итоговая сумма пересчитана
```

### Ключевое правило

Сумма к возврату рассчитывается системой автоматически из состава заказа.
Ни клиент, ни Елена не должны считать вручную.

## 12. Шоурум-визит

### Ценность визита в шоурум

Клиент физически примерил вещи — это самые ценные данные.
Он уже знает, что подходит. Нет риска возврата по размеру.
Система должна превратить одну встречу в длинные продажи.

### Сценарий визита

```
1. Клиент приезжает в шоурум
2. Елена открывает его профиль на планшете/телефоне
3. В процессе примерки каждая вещь отмечается:
   💛 "Нравится, но не сейчас" → вишлист с тегом "примеряла"
   ✅ "Беру" → уходит в заказ
   ❌ "Не подошло" → отмечается с причиной
4. По окончании визита профиль клиента обновлён
5. Клиент уезжает — в кабинете уже всё зафиксировано
```

### Что видит Елена после визита

```
СЕССИЯ ПРИМЕРКИ: Людмила, 15 марта 2025

КУПИЛА (2 вещи) → ORD-2025-004
  ✓ Платье Art-214 (беж, 46)
  ✓ Палантин Art-07

ХОЧЕТ, НО НЕ СЕЙЧАС (вишлист — 5 вещей):
  💛 Кардиган Art-31 (молочный, 46) — "идеально"
  💛 Юбка Art-112 (беж, 46) — "ждёт зарплату"
  💛 Платье Art-301 (терракот, 46) — "может быть"
  → [Запустить акцию для них]  [Написать клиенту]

НЕ ПОДОШЛО (не предлагать снова):
  ❌ Костюм Art-077 — велико в плечах
  ❌ Платье Art-188 — слишком яркое
```

### Тактика после визита

```
Было: клиент приехал → купил 2 вещи → ушёл → забыл
Стало: клиент приехал → примерил 18 → купил 2 → ушёл
       → система помнит 5 вещей, которые подошли и понравились
       → Елена продаёт их в течение следующих месяцев
       → без повторного объяснения размеров
       → без риска возврата по посадке
```

## 15. Голос Елены

### Главный принцип

Автоматизация не должна убивать тепло.
Клиент должен чувствовать, что написала Елена — не рассылочный робот.

### Как строится персональное сообщение

```
Блок 1. Обращение
  → имя клиента
  → тёплое вступление (не "уважаемая", а "Людмила, добрый день!")

Блок 2. Личная привязка
  → что стоит в вишлисте / что примеряла / что искала
  → или: что она говорила ("вы искали на свадьбу подруги")
  → или: событие из профиля ("у вас в июне свадьба")

Блок 3. Товар и почему именно ей
  → что пришло / что со скидкой
  → почему подходит: цвет волос, тон кожи, предпочтения, повод

Блок 4. Личное от Елены
  → "мне кажется, это прям ваше"
  → "сразу подумала о вас, когда увидела"
  → "уверена — будете в нём королевой"

Блок 5. Действие
  → ссылка на карточку товара
  → "напишите мне, если хотите — отложу для вас"
```

### Пример персонального сообщения

**Данные из профиля:** волосы тёмные, в вишлисте платье-миди, событие — свадьба подруги в июне, личная фраза — "не хочу броского, хочу элегантно"

> Людмила, добрый день!
>
> Помню, вы говорили, что ищете что-то элегантное на свадьбу подруги в июне — не броское, но чтобы запомнилась.
>
> К нам только что пришло платье — миди, свободный верх, очень благородный пыльно-розовый. Мне кажется, под ваши тёмные волосы будет очень красиво — как раз тот тон, который освежает и не кричит.
>
> Посмотрите — [ссылка]. Если хотите, отложу для вас, пока есть ваш размер.
>
> Елена

### Как работает процесс отправки

```
1. Система видит: новый товар похож на вишлист Людмилы
2. ИИ берёт данные из профиля и составляет черновик
3. Елена видит черновик в интерфейсе
4. Правит 1–2 слова / одобряет
5. Нажимает "Отправить"
```

Елена не пишет с нуля. Она правит готовое.
Это экономит время и сохраняет её голос.

### Голос Елены в системе (банк фраз)

Елена один раз добавляет свои типичные фразы:

```
Вступления:
  "Людмила, добрый день!"
  "Добрый день! Сразу подумала о вас..."

Личное:
  "мне кажется, это прям ваше"
  "сразу подумала о вас, когда увидела"
  "очень идёт / очень освежает"
  "будете в нём королевой"
  "уверена — понравится"

Действие:
  "отложу специально для вас"
  "напишите мне"
  "посмотрите — [ссылка]"
```

ИИ использует эти фразы при составлении черновиков — не придумывает чужой стиль.

### Ключевое правило

```
❌ Нельзя: кнопка "Разослать всем" с одним текстом
✅ Можно: система готовит черновик для каждого клиента отдельно
          Елена одобряет и отправляет каждому
```

## 17. План запуска

### Этап 0 — Подготовка (1–2 недели)

- Зафиксировать все активные заказы в таблице
- Составить список клиентов с параметрами
- Согласовать с Еленой: она готова работать в новой системе?
- Выбрать технологию (no-code или разработка)

### Этап 1 — MVP (4–8 недель)

Запускается:
- Каталог товаров (ввод вручную / импорт CSV)
- Карточка клиента + форма заполнения
- Карточка заказа + позиции + статусы
- Ссылка на заказ для клиента (только чтение)
- Форма запроса/подбора для клиента
- Базовый вишлист

### Этап 2 — Логистика (3–4 недели после этапа 1)

- Сценарий частичной отправки (посылки внутри заказа)
- Трек-номера в карточке заказа
- Уведомление клиенту о составе посылки

### Этап 3 — Возвраты (2–3 недели)

- Форма возврата
- Автоматический расчёт суммы к возврату
- Статусы возврата

### Этап 4 — Шоурум и вишлист (3–4 недели)

- Сессия примерки в шоуруме
- Расширенный вишлист (с тегом "примеряла")
- Уведомления об акциях на вишлист (email / WhatsApp)

### Этап 5 — Рекомендации (4–6 недель)

- Связи между похожими товарами
- Инструмент "продвинуть товар похожим клиентам"
- ИИ-черновики персональных сообщений
- Банк фраз голоса Елены

### Этап 6 — Аналитика (после стабилизации)

- Какие товары чаще возвращают
- Какие клиенты наиболее ценные
- Средний чек, конверсия
- Спрос по вишлисту → приоритет при закупке

## Итого: система в одной схеме

```
Поставщики/склады
(фото, видео, таблицы)
        ↓
КАТАЛОГ ЕЛЕНЫ
(классика + новинки + фильтры)
        ↓
КЛИЕНТ
├── Новый → Анкета → Профиль → Форма запроса → Подборка
├── Постоянный → Кабинет → Мои вещи → Повторить заказ
└── После шоурума → Примерочный вишлист → Уведомления

ЗАКАЗ
└── Фиксирован → Статус → Посылки + треки → Возврат → Закрыт

МАРКЕТИНГ
└── Вишлист → Триггер → ИИ-черновик → Елена правит → Отправка
    (не спам, а персональная забота с голосом Елены)
```
